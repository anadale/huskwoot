# Спецификация: инструмент update_task

> **Статус:** Проектирование.

## Цель

Добавить в агент инструмент `update_task`, позволяющий через DM или упоминание бота изменить
любые редактируемые свойства задачи: краткое описание, детали, дедлайн, статус.

---

## Контекст

Уже существуют инструменты `complete_task` и `move_task`. `update_task` следует тому же
принципу разделения ответственности:

**`update_task` не занимается поиском.** Инструмент принимает только UUID задачи (`task_id`).
Нахождение нужного UUID — задача LLM-агента, который использует `list_projects` и `list_tasks`
для поиска, а при неоднозначности предлагает пользователю выбрать из вариантов.

---

## Требования

### Идентификация задачи

`update_task` принимает единственный идентификатор:

| Параметр | Тип | Описание |
|---|---|---|
| `task_id` | string (UUID) | Обязательный. UUID задачи, полученный из `list_tasks`. |

Если `task_id` не передан — ошибка.

### Как агент находит UUID

Агент (LLM) должен самостоятельно разрешить ссылку пользователя в UUID задачи:

1. **По части названия:** вызвать `list_tasks` с `query` = ключевые слова.
2. **По номеру задачи в проекте** (например, пользователь говорит «inbox#5» или «задача #5 в Работе»):
   - Если проект неизвестен — вызвать `list_projects`, найти нужный проект.
   - Вызвать `list_tasks` с `project_id` найденного проекта (без query или с небольшим query).
   - Найти задачу с нужным `display_id` (`<slug>#<number>`) в результате.
3. **Однозначный результат** (ровно одна задача) → использовать её `id` как `task_id`.
4. **Несколько совпадений** → вывести пользователю пронумерованный список первых 3 задач
   (`#ID: summary`) и попросить уточнить: «Уточни номер задачи: 1, 2 или 3?». Дождаться ответа,
   затем вызвать `update_task` с нужным `task_id`.
5. **Ноль совпадений** → сообщить пользователю, что задача не найдена.

Это поведение описывается в системном промпте агента и не реализуется в коде инструмента.

### Обновляемые поля

| Параметр | Тип | Поведение |
|---|---|---|
| `summary` | string | Новое краткое описание (пустая строка — нет изменений) |
| `details` | string | Новые детали; передать `""` чтобы очистить |
| `deadline` | string | RFC3339, русское выражение («завтра», «в пятницу») или `"none"` для снятия срока |
| `status` | string | `open`, `done`, `cancelled` (пустая строка — нет изменений) |

Если параметр не передан или является пустой строкой (за исключением `details` и `deadline`
со специальной семантикой) — соответствующее поле задачи не изменяется.

### Семантика deadline

- Не передан → `TaskUpdate.Deadline = nil` (без изменений).
- `"none"` или `""` → `TaskUpdate.Deadline = &(*time.Time)(nil)` (снять срок).
- Любое другое значение → разобрать через `dateparse.Parse(value, now, cfg)`;
  при ошибке — вернуть ошибку инструмента.

### Семантика details

- Не передан → `TaskUpdate.Details = nil` (без изменений).
- `""` (пустая строка) → `TaskUpdate.Details = ptr("")` (очистить поле).
- Любое другое значение → `TaskUpdate.Details = ptr(value)`.

Примечание: для `summary` и `status` пустая строка означает «без изменений», а не очистку,
так как пустое summary и пустой статус не имеют смысла.

### Исключения

- Номер задачи (`number`) не изменяется.
- Перенос задачи в другой проект — отдельный инструмент `move_task`.

## Архитектура

### Компонент

Новый файл `internal/agent/tool_update_task.go`:

```go
type updateTaskTool struct {
    tasks model.TaskService
    cfg   dateparse.Config
}
```

### Конструктор

```go
func NewUpdateTaskTool(tasks model.TaskService, cfg dateparse.Config) Tool
```

### Tool-интерфейс

| Метод | Значение |
|---|---|
| `Name()` | `"update_task"` |
| `Description()` | `"Изменить свойства задачи: описание, детали, дедлайн, статус"` |
| `DMOnly()` | `false` |
| `Parameters()` | JSON-схема (см. ниже) |

### JSON-схема параметров

```json
{
  "type": "object",
  "properties": {
    "task_id":  { "type": "string", "description": "UUID задачи (получить из list_tasks)" },
    "summary":  { "type": "string", "description": "Новое краткое описание (не изменяется если пустая строка)" },
    "details":  { "type": "string", "description": "Новые детали (пустая строка очищает поле)" },
    "deadline": { "type": "string", "description": "Новый срок: RFC3339, русское выражение или \"none\" для снятия" },
    "status":   { "type": "string", "description": "Новый статус: open, done, cancelled" }
  },
  "required": ["task_id"]
}
```

### Логика Execute

```
1. Разобрать JSON-аргументы
2. Проверить наличие task_id; если пусто → ошибка
3. GetTask(ctx, task_id) → если не найдена → ошибка
4. buildUpdate(params, now):
   - summary: ptr(value) если value != ""
   - details: ptr(value) всегда если поле передано в JSON (nil если отсутствует)
   - deadline: "none"/"" → ptr(nil); иначе dateparse.Parse
   - status: ptr(value) если value != ""
5. TaskService.UpdateTask(ctx, task_id, update)
6. Вернуть JSON с обновлёнными полями задачи
```

### Ответ инструмента (успех)

```json
{
  "id":         "<UUID>",
  "display_id": "<slug>#<number>",
  "summary":    "...",
  "details":    "...",
  "deadline":   "2026-05-01T00:00:00Z",
  "status":     "open"
}
```

## Изменения системного промпта

В `internal/agent/prompts/agent_system.tmpl` добавить раздел:

```
Изменение задачи (update_task):
- update_task принимает только task_id — UUID задачи из поля "id" в результате list_tasks.
- Если пользователь называет задачу по части названия — используй list_tasks с параметром query для поиска нужного task_id.
- Если пользователь называет задачу по номеру (например «inbox#5» или «задача #5 в Работе»):
  1. Если проект неизвестен — вызови list_projects и найди нужный проект.
  2. Вызови list_tasks с project_id этого проекта; найди задачу с нужным display_id в результате.
- Если найдена ровно одна задача — вызывай update_task с её id.
- Если найдено несколько — выведи первые 3 в виде пронумерованного списка и спроси: «Уточни: 1, 2 или 3?». Дождись ответа и вызови update_task с нужным id.
- Если задача не найдена — сообщи об этом пользователю и не вызывай update_task.
```

Покрываемые сценарии (11 кейсов):

2. Установка дедлайна через RFC3339

9. Некорректный JSON — ошибка
10. `task_id` не задан — ошибка
11. Метаданные: `Name() == "update_task"`, `DMOnly() == false`

`updateTaskResult`, `updateTaskErr`, `lastUpdateID`, `lastUpdateUpd`.
